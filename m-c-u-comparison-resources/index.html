FPGA 的配置存储与架构机制

我们在做大功率控制板时，很多时候会把 FPGA 当作逻辑协处理器使用。

很多刚入行的工程师对 FPGA 的程序存储总有误解。其实，大部分主流 FPGA（如 Xilinx 7系列、Altera Cyclone）内部的逻辑配置单元都是基于 SRAM 技术的。SRAM 是一种易失性存储器，这就意味着一旦断电，芯片内部的硬件配置映射图（即比特流 Bitstream）就会全部丢失。

为了解决这个问题，我们在画板子时，必须在 FPGA 旁边外挂一片非易失性的 SPI Flash 芯片：

  • 开发阶段：我们在 Vivado 或 Quartus 中完成 Verilog 代码编写，生成定义硬件电路连线的比特流文件（.bit 或 .pof）。
  • 烧录阶段：通过 JTAG 调试器，把比特流烧录到板载的外部 Flash 芯片中。
  • 上电加载：每次重新上电时，FPGA 内部固化的加载状态机（如 Master SPI 模式）会自动从外部 Flash 中把比特流读取出来，加载到自己的 SRAM 中。这个加载过程一般在几十毫秒到几百毫秒内完成，加载成功后，FPGA 才正式投入运行。

这个过程就像电脑开机时，主板从硬盘（Flash）里把操作系统读入到内存（SRAM）中跑一样。

那些不需要外部 Flash 的 FPGA 类型：

当然，我们在一些特定场景下也会用到这三种例外：

  • 基于 Flash 技术的 FPGA (Flash-based FPGA)：比如 Microchip 的 ProASIC3 或者 Lattice 的 MachXO2 系列。它们的配置单元天生就是 Flash，程序直接烧在芯片里，上电瞬间就能跑（Instant-on），抗辐射安全性极高，但代价是内部可用的逻辑资源和运行主频往往不如 SRAM 型 FPGA。
  • SoC FPGA：像 Xilinx 的 Zynq-7000 或 Intel 的 Cyclone V SoC。这类芯片在单颗硅片上集成了 ARM 双核硬核（PS）和 FPGA 可编程逻辑（PL）。上电后，ARM 处理器先启动，从 QSPI Flash 或 SD 卡加载 Linux 等操作系统，然后再由系统软件动态地把比特流写进 PL 区完成配置。
  • 主控遥控配置：如果控制板上有更强的主控（如 ARM 或 x86），我们可以让主控直接通过 JTAG 或 SPI 总线，把比特流灌进 FPGA 里完成配置。

理论计算变比 vs 产线实测真值

在实际开发中，我们要严格区分这两个概念：

  • 理论变比：基于原理图上的分压电阻阻值、互感器圈数等 ideal 状态算的理论值。这是我们写初版代码时的“瞎猜值”，用来验证通路是否正常。
  • 物理标准真值：我们在设备出厂前，使用 6 位半万用表实测出来的物理量。这是为了消灭电阻公差（如 1% 精度电阻）、温漂、运放输入失调偏置和 ADC 转换公差的真实参数。
维度 硬件理论变比 两点校正标准真值
计算依据 原理图上的标称元件参数公式。 6位半高精度万用表或高精度源实测值。
涵盖误差 忽略所有硬件公差与寄生效应。 完美补偿阻容公差、运放失调、ADC 基准温漂等系统误差。
使用时机 研发初期编写演示代码时。 出厂前的自动化 ATE 测试、高性能闭环控制。

ATE 自动化测试流水线校准流程

在大规模量产时，人工用表笔点着去测参数简直是灾难。我们都是采用 ATE（Automated Test Equipment） 自动化测试台架来标定：

  1. 系统架构：中控台 PC 运行自动化测试脚本，通过 GPIB/USB 控制高精度交流/直流电源，并通过串口/CAN 连接待测的控制板。
  2. 测试流程：
  • 主控 PC 控制电源输出指定精度的 0V 和 24V 激励；
  • PC 读取 6位半万用表，得到标准真值 和 ；
  • PC 通过串口向下位机 MCU 发送查询指令，读回此时对应的 ADC 采样值 和 ；
  • 主控 PC 脚本自动计算出 和 ，然后通过串口协议发送给 MCU；
  • MCU 将这两个系数永久写入非易失性存储器（如内部 Flash 模拟 EEPROM，或板载的物理 EEPROM 中）。

研发阶段与量产阶段的存储策略差异

在设计存储策略时，很多新手一上来就照抄量产方案，结果在调试样机时把自己坑惨了：

  • 量产阶段（EEPROM / Flash 备份，推荐）：这是标准做法。下线时的每台设备都跑同一套二进制固件。上电时，MCU 的驱动读取特定 EEPROM 扇区的标定参数，如果有则覆盖默认理论值；如果在升级时重刷固件，由于标定参数存在独立的 EEPROM 扇区，因此参数不会被覆盖。
  • 研发打样阶段（代码硬编码，推荐）：在研发初期调试一两台样机时，千万别搞复杂的物理存储标定流程！
    • 因为这时候你的 I2C 驱动、SPI 驱动或者 Flash 读写逻辑还没调稳，很容易读出全是垃圾字符的脏数据，导致采样出现离奇误差。
    • 更实用的办法：直接把实测计算出来的 和 作为常量，硬编码写死在软件头文件的宏定义里。这样可以排除所有外部通信的干扰，让我们集中精力调试核心算法。

C2000 CSM（代码安全模块）的工作原理与意外锁死排查

C2000 的 CSM (Code Security Module) 是为了防抄板、保护我们辛苦写的控制算法而设计的硬件密码锁。

在 F28035、F280049 等芯片中，Flash 的特定末端地址处预留了一组 128 位的密码位（Password locations）。

  • 出厂状态：默认全为 0xFFFF，此时芯片处于解锁状态，你可以随意用 JTAG 擦写和读写。
  • 上锁状态：一旦我们向这些特定密码地址写入了任何非 0xFFFF 的数据，芯片在上电后就会自动锁死。
  • 解锁：在上锁状态下，如果你要通过 JTAG 读写 Flash，必须在 CCS 的 Debug 界面向特定的 CSM 密钥寄存器写入跟 Flash 密码完全一样的 128 位密码，否则仿真器会读取到一片全 0，并且无法进行任何擦除和编程操作。

导致芯片“莫名其妙”锁死的常见死因：

在研发阶段，我们经常会遇到芯片被意外锁死报废的情况，这通常不是我们故意设了密码，而是踩了以下几个坑：

  1. Linker Link 链接文件配置错误 (.cmd)：很多兄弟直接从网上拷 .cmd 文件。如果链接文件里，把用户代码的某些段（如常量、初始化数据）错误地分到了包含 CSM 密码区地址的 Flash 物理扇区，烧录程序时，编译器就会把你的代码直接当成“密码”写了进去。拔掉仿真器重新上电后，芯片当场被这串随机的代码锁死，且你无法知道它的实际数据。
  2. Flash 烧写过程意外掉电：在 Flash 执行擦除或烧写瞬间，如果控制板的 3.3V 辅助电源突然发生打火、跌落，或者调试线松动连接中断，会导致 Flash 密码区的浮栅电荷处于半悬空状态。硬件判定其为非全 1 状态，芯片直接永久锁死。
  3. 指针跑飞误写：在软件调试时，如果由于数组越界或者内存写溢出，导致指针跑飞，误向 Flash 密码区写入了垃圾数据。

仿真器质量与虚假锁死 (Error -1015) 的排查

  • 劣质仿真器的危害：市面上那些几十块钱的克隆版 JTAG 调试器，不仅没有隔离，而且它的驱动波形存在明显的抖动（Ringing）和幅值不够。在擦写 Flash 等对时序要求极高的时间窗内，一旦时钟线 TCK 出现毛刺，就会引发 Flash 控制状态机逻辑混乱，从而把垃圾数据写进密码区把芯片锁废。
  • Error -1015 的真假虚实：CCS 报错 Error -1015 (Device is locked or not responding) 时，先别慌，这不代表芯片一定真的锁死了！
    • 复位死循环假象：如果你的代码里有时钟配置错误，或者一开机看门狗就超时，导致芯片以几百微秒的频率不停地进行硬件复位。此时仿真器根本没法跟处于复位中的芯片握手，CCS 就会误报锁死。
    • 救活方法：把板子上的引导模式拨码开关拨到 Wait Boot 模式（GPIO72=0, GPIO84=1）。上电后，Boot ROM 会在死循环里一直挂起，不会去执行 Flash 里的损坏代码。此时再连仿真器，通常就能连上，然后用 UniFlash 把 Flash 全片擦除即可救活。

TI C2000 保留这种高风险的 CSM 设计，很大程度上是为了兼容几十年来在工业领域（如高压变频器）的汇编代码。同时工业级安全理念认为：一旦内部 Flash 被意外改写，说明芯片已进入非安全状态，必须永久锁死，防止“带病运行”导致大功率电力变换桥短路炸机。

C2000 的实时波形录制（录波）资源消耗

在数字电源和电机控制调试中，我们经常需要在 RAM 中开辟空间进行高速录波（类似于软件示波器），去观察动态波形（如启动瞬间的瞬态电流）。

我们的解决方法：乒乓缓存（Ping-Pong Buffering）

我们会在 RAM 里开辟两个交替的缓冲区（Buffer A 和 Buffer B）。

  • 当下半个周期 ADC 把数据填满 Buffer A 时，触发 DMA 或高速中断，将 Buffer A 的数据通过 UART / CAN 或以太网发送到上位机；
  • 此时 ADC 自动切换去填充 Buffer B；
  • 发送完毕后，Buffer A 清空，等 Buffer B 满时再切回来。
    这种乒乓接力的方式，能让我们用极小的 RAM 缓冲空间实现长达数秒的高保真波形录制。

TI C2000 第三代产品（Gen 3）选型指南

TI 的 C2000 目前推荐在新项目中全部采用第三代（Gen 3）芯片。我们把几个主流系列的定位和典型场景总结如下：

核心应用场景与推荐：

  1. 高频数字电源（服务器 CRPS 电源、图腾柱 PFC、LLC）
    • 推荐型号：TMS320F280039C 或 F280049C
    • 理由：内置 Type-4 PWM 模块和高分辨率 PWM（HRPWM，步进精度高达 150ps），是跑 MHz 级别宽禁带半导体（GaN/SiC）控制的必备利器。
  2. 新能源汽车（车载 OBC、高压 DCDC、牵引逆变器）
    • 推荐型号：F280039C-Q1 (车规版) 或双核的 F2837xD / F2838x
    • 理由：符合 ISO 26262 车规功能安全（ASIL-B），内部集成 CLB（可配置逻辑块），可以省去板子上的小 CPLD。如果是汽车双电机牵引控制，直接上双核跑控制，另一个核做监控和通信。
  3. 伺服电机驱动器（多轴协同、精密控制）
    • 推荐型号：TMS320F280049C 或 F28377D
    • 理由：芯片 ROM 里直接固化了 InstaSPIN-FOC 算法（含 FAST 估算器），三分钟就能测出电机的电感电阻参数。内置的 TMU（三角函数加速器），可以在 内算完 FOC 环路的 Park / Clark 变换和正余弦。

异构多核架构深度解析：C28x 与 ARM 的协同之道

在高端通信型旗舰（如 F2838x）中，TI 引入了 “C28x + ARM” 的异构多核架构。

这是为了解决传统控制板上“用一颗 DSP 既跑 100kHz 的 PWM 控制，又跑复杂的 TCP/IP 以太网协议栈，导致系统时钟抖动和实时性崩盘”的问题。

  • C28x 内核：硬实时控制专家（The Controller，肌肉）
    • 指令周期确定，中断响应极快（ns级），内置 TMU 指令集。专职运行 FOC 电流环、高频电源占空比计算和硬件故障切断。
  • ARM Cortex-M4：通信管理专家（The Manager，大脑）
    • 拥有海量的第三方库生态。专门运行 EtherCAT、TCP/IP、USB DFU 固件升级和系统逻辑调度。

调试仿真器选型：XDS100、XDS110 与 XDS200 的深层区别

在做 C2000 开发时，仿真器是我们最常用的硬件伙伴。

经常有师弟问我，这三代调试器到底差在哪里？我们整理如下：

  • XDS100v2 / v3（入门级，已淘汰）：内部采用 FTDI 芯片做 USB 转串口协议转换，速度只有 USB Full Speed（12Mbps）。烧录大代码时慢得像蜗牛。在开启 Real-time 模式调试时，变量刷新率极低，根本无法实时观察电机的 PI 环路收敛过程。
  • XDS110（主流标配，性价比高）：基于 ARM 单片机方案，支持 USB 2.0 High Speed（480Mbps）。烧录速度比 XDS100 快 3 到 5 倍。我们在日常开发中，它的变量刷新刷新率完全能撑起我们对 PID 调节参数的观察。
  • XDS200（高端旗舰，极速调试）：采用 FPGA 方案，时钟频率高，下载大型固件基本都是秒级完成，调试断点极度灵敏，唯一的缺点是原装价格较贵。

电力电子高压调试的安全红线：自制隔离版 XDS110

炸机波及电脑风险：我们在调试 400V 甚至 800V 高压母线电源时，大功率管一旦发生雪崩击穿“炸管”，几百伏的高压会顺着主控地线窜入控制板的 JTAG 调试口，然后流进仿真器，最终把你的笔记本电脑主板和 USB 端口烧毁。

防范对策：我们必须使用带电气隔离（ISO）的仿真器。
我们在研发中，通常会自己用高速磁耦隔离芯片（如 ADUM1201 + ADUM1401）和隔离电源模块（如 5V 转 5V 的隔离 DCDC），自制带 2.5kV 隔离的 JTAG 下载板。它的安全价值，在强电环境下甚至远高于非隔离的 XDS200。

板载调试核心：FT2232 硬件 ASIC 方案 vs XDS110 方案

为了方便生产和用户连接，我们经常在产品控制板上直接集成板载调试器，最常用的就是 FT2232 方案和 XDS110 方案。

对比项 FT2232 板载调试芯片方案 XDS110 单片机调试方案
主控类型 纯硬件 ASIC 专用桥接芯片。 运行在 120MHz 的 Cortex-M4 微控制器。
通信协议机制 电脑端 CPU 运行 JTAG 时序，通过 FT2232 物理转换。 仿真器单片机内部运行协议栈，本地生成时序波形。
烧写校验速度 受到 USB 通信的往返延迟制约，校验较慢。 内部有大内存缓冲，本地校验，下载极快。
固件更新 芯片纯固化逻辑，终生免维护，不需要升级。 随 TI CCS 软件升级，经常需要弹窗联机更新固件。
串口集成 自带硬件双通道，A路调 JTAG，B路调串口通信。 由单片机程序虚拟出的串口，稳定性略逊。

生产烧写流程：

  • FT2232 身份注入：初次贴片上电时，FT2232 只是普通芯片。我们需要连接电脑，用配置工具改写它外挂的 EEPROM 数据，将其 VID 设为 0403，PID 设为 A6D0。重插后，电脑就会将其判定为官方调试器，不需要编写任何代码。
  • XDS110 系统引导：上电时芯片是空的。我们必须用另一台外部下载器，通过 SWD 接口给 XDS110 的 MCU 烧写底层的 Bootloader 引导程序；然后运行 TI 命令行 DFU 工具，把包含实时操作系统和 JTAG 协议栈的完整固件写入。

处理器架构深度对比：CPU/DSP vs FPGA 在高频控制中的博弈

大功率控制中的核心差异，在于中断跳转与并行流水线的执行机制。

  • CPU/DSP (C2000) 的局限：不管主频多高，它本质上还是一个串行处理器。
    • 处理 FOC 环路时，ADC 完成采样 -> 触发外部中断 -> CPU 保存当前寄存器（压栈） -> 跳转入 ISR 中断函数 -> 顺序计算 PI -> 将占空比写进 PWM 寄存器 -> 恢复寄存器（出栈）。
    • 进出中断的固定开销（Overhead）和若有更高优先级中断抢占产生的时钟抖动（Jitter），决定了它的闭环频率上限极难做到 500kHz 以上。
  • FPGA 的数据流控制：内部是纯粹由逻辑门构成的并行硬件电路。
    • 它不需要任何中断。一旦 ADC 模块数据准备好，直接拉高握手信号 adc_valid，数据瞬间流入物理存在的 PI 运算乘加器电路，在短短几十纳秒（ns）内完成运算，数据直通写入 PWM 活动寄存器。
    • 这种并行执行无任何打断概念，延迟极低，因此是超高频 GaN 电源（开关频率 >1MHz）和 MMC 模块化多电平换流器（需要同步控制几百个子模块）的唯一解。

高精度 PWM 同步更新与影子寄存器机制

不管在 DSP 还是 FPGA 里，如果你的 PI 计算在周期中途算完就立即更新占空比，会导致生成的 PWM 波形严重不对称，从而产生逆变电路的偏磁饱和。

  • 正确做法：使用影子寄存器（Shadow Register）。
  • 控制算法算出的新占空比，在周期中途写入影子寄存器（缓冲）；
  • 硬件自动等待定时器计数到周期值（归零或峰值点，即 CTR=0 或 CTR=PRD）；
  • 硬件在一个时钟周期内，把影子寄存器的值加载到活动寄存器中，在下一个周期统一生效，从而保障了输出波形的绝对对称。

三大微控制器平台硬核对比：TI C2000 vs ARM (STM32) vs Infineon AURIX

这三者在工业和汽车电子控制中，分别代表了三个方向的设计精髓：

特性维度 STM32G4 (ARM Cortex-M4) TI C2000 (C28x DSP) Infineon AURIX (TriCore)
定位精髓 工业通才，高性价比与生态平衡。 极致效能，为实时数学运算而生的“特种兵”。 车规安全堡垒，确保极端故障不损坏系统。

位操作能力 较弱，常规操作需要三步读改写流程。 极强，支持硬件级原子操作，GPIO翻转极快。 极强，支持强大的位域提取和插入，适合CAN。
三角加速 CORDIC 协处理器。作为总线上的独立外设，CPU 需写入数据 -> 等待计算 -> 读回。 TMU 单元。直接集成在 CPU 核心内，sin/cos 会被自动编译为 1-3 周期汇编指令。 SPU 信号处理单元，依靠专用指令实现。

处理器字宽与指令级硬件架构的本质

  • 处理器的位数：是指 CPU 通用寄存器、数据总线和地址总线单次操作的数据位宽。32位微控制器单次可以吞吐 32 位的整型数据，并拥有 4GB 的直接寻址空间。
  • 哈佛架构（C2000 C28x） vs 冯诺依曼架构：C28x 采用独立的程序和数据总线（哈佛架构），可以让 CPU 一边取控制指令，一边去 RAM 中取采样数据；而普通的 ARM 系列虽然也支持类似总线设计，但底层总线竞争仍然存在，导致在执行多路 PI 累加时效率弱于专为 DSP 优化的哈佛总线。
  • Thumb-2 指令集（ARM）：它能在 16 位和 32 位指令长度之间动态切换，获得了极高的代码密度（Code Density），相比传统的 32 位 ARM 指令，能够省下 30% 以上的 Flash 存储空间，这也是 STM32 协议栈跑得很流畅的原因。
